    1. Objetivo:

      1. O TSDoc uma proposta para padronizar os comentrios de documentos usados no cdigo TypeScript , para que diferentes ferramentas possam extrair contedo sem se confundir com a marcao umas das outras. Mais informaes sobre a especificao tsdoc veja aqui.

      2. Com os comentrios padronizados baseado em TSDoc o programa TypeDoc gera um site inteiro com a documentao de todas as classes e funes.

  3. Descrio

    1. O pacote @microsoft/tsdoc a implementao de referncia de um analisador para a sintaxe TSDoc. Voc no pode us-lo diretamente. um componente do mecanismo que deve ser incorporado a outras ferramentas de documentao.

    2. Se voc estiver implementando uma ferramenta que precisa extrair informaes dos comentrios do cdigo TypeScript, @microsoft/tsdoc fornece uma soluo fcil que implementar corretamente o TSDoc.

    3. A anatomia geral de um comentrio TSDoc tem estes componentes:

      1. A seo de resumo: O contedo da documentao at a primeira tag de bloco chamado de resumo. A seo de resumo deve ser breve. Em um site de documentao, ele ser mostrado em uma pgina que lista resumos de vrios itens de API diferentes. Em uma pgina de detalhes para um nico item, o resumo ser mostrado seguido pela seo de comentrios (se houver).

      2. O bloco de comentrios: O bloco de comentrios comea com a tag @remarks do bloco. Ao contrrio do resumo, as observaes podem conter um contedo de documentao extenso. A seo de comentrios no deve reformular as informaes do resumo, uma vez que a seo de resumo sempre ser exibida onde quer que a seo de comentrios aparea.

      3. Blocos adicionais: Outros blocos TSDoc normalmente seguem o @remarks bloco. Cada bloco introduzido por uma etiqueta de bloco, tais como @param, @returns, @example, etc.

      4. Tags modificadoras: as tags modificadoras geralmente vm por ltimo. As tags modificadoras no tm um bloco de contedo associado; em vez disso, sua presena indica um aspecto da declarao. Alguns exemplos de marcas modificadoras so: @public, @beta, e @virtual.

      5. Tags embutidas: as tags embutidas podem aparecer dentro de qualquer seo e so sempre delimitadas pelos caracteres { e }. O contedo adicional pode aparecer aps o nome da tag e antes do } delimitador. Seu formato especfico da tag. Exemplos de tags embutidas so {@link alvo | ttulo do alvo} e {@inheritDoc}.

    4. Tags de liberao:

      1. Os quatro tags de lanamento so: @internal, @alpha, @beta, @public. Eles so aplicados a itens de API, como classes, funes de membro, enums, etc. O API Extractor classifica cada item de API exportado individualmente, de acordo com seu nvel de suporte pretendido:

        1. internal: indica que um item de API deve ser usado apenas por outros pacotes NPM do mesmo mantenedor. Terceiros nunca devem usar APIs internas. Para enfatizar isso, os prefixos de sublinhado devem ser usados para itens com uma tag @internal (explcita).

        2. alpha: indica que um item de API eventualmente se destina a ser pblico, mas atualmente est em um estgio inicial de desenvolvimento. Terceiros no devem usar APIs alfa.

        3. beta: indica que um item da API foi lanado como uma visualizao ou para fins experimentais. Os terceiros so incentivados a experiment-lo e fornecer feedback. No entanto, uma API beta NO deve ser usada na produo, pois pode ser alterada ou removida em uma verso futura.

        4. public: indica que um item de API foi lanado oficialmente e agora faz parte do contrato com suporte de um pacote. Se o esquema de verso SemVer for usado, a assinatura da API no poder ser alterada sem um incremento de verso MAJOR.

      2. Quando uma API introduzida pela primeira vez, geralmente comea como @alpha. Conforme o design amadurece, ele se gradua em @alpha -> @beta -> @public . A designao @internal usada principalmente para resolver problemas de encanamento e geralmente no est em nenhum roteiro para se tornar pblica. (Tambm existe uma tag @deprecated, mas uma opo que pode ser combinada com qualquer uma das tags acima.)

      3. A tag de liberao se aplica recursivamente aos membros de um continer (por exemplo, class ou interface). Por exemplo, se uma classe for marcada como @beta, todos os seus membros tero automaticamente esse status; voc NO precisa adicionar a tag @beta a cada funo de membro. No entanto, voc pode adicionar @internal a uma funo de membro para dar a ela um status de liberao diferente.

      4. Por ltimo, observe que certas regras lgicas se aplicam. Por exemplo, uma @public funo no deve retornar um tipo @beta. Uma classe @beta no deve herdar de uma classe base @internal, etc. Atualmente, o API Extractor no valida essas regras, mas o far em breve.

  4. As tags TSDoc abaixo so usadas dentro de um comentrio (/** */) precedida do caractere @. Quando precisar que @tag seja visualizada no texto ento coloca-la entre o caractere ( ` crase). Exemplo: `@tag`.

    1. TSDOC JSDOC DESCRIO
      @alpha x Designa que o estgio de lanamento de um item de API alfa. Destina-se a ser usado por desenvolvedores de terceiros eventualmente, mas ainda no foi lanado. O ferramental pode cortar a declarao de uma liberao pblica. Exemplo...
      @beta x Designa que o estgio de lanamento de um item de API beta. Ele foi lanado experimentalmente para desenvolvedores terceirizados com o objetivo de coletar feedback. O API no deve ser usado na produo, pois seu contrato pode ser alterado sem aviso prvio. A ferramenta pode cortar a declarao de uma verso pblica, mas pode inclu-la em uma verso de visualizao do desenvolvedor. Exemplo...
      @decorator x Os decoradores ECMAScript s vezes so uma parte importante de um contrato de API. No entanto, hoje o compilador TypeScript no representa decoradores nos arquivos de sada .d.ts usados por consumidores de API. A @decorator tag fornece uma soluo alternativa, permitindo que uma expresso do decorador seja citada em um comentrio de documento. Exemplo...
      @default Value @default Esta tag de bloco usada para documentar o valor padro para um campo ou propriedade, se um valor no for atribudo explicitamente. Essa tag s deve ser usada com campos ou propriedades que so membros de um TypeScript class ou interface. Exemplo...
      @deprecated @deprecated Esta tag de bloco comunica que um item de API no mais compatvel e pode ser removido em uma verso futura. A @deprecated tag seguida por uma frase que descreve a alternativa recomendada. Ele se aplica recursivamente aos membros do continer. Por exemplo, se uma eventproperty classe for descontinuada, todos os seus membros tambm sero. Exemplo...
      @event Property @event Quando aplicada a uma classe ou propriedade de interface, indica que a propriedade retorna um objeto de evento ao qual os manipuladores de eventos podem ser anexados. A API de manipulao de eventos definida pela implementao, mas normalmente o tipo de retorno de propriedade seria uma classe com membros como addHandler()e removeHandler(). Uma ferramenta de documentao pode exibir essas propriedades sob um ttulo Eventos em vez do ttulo Propriedades usual. Exemplo...
      @example @example No Tsdoc indica uma seo de documentao que deve ser apresentada como um exemplo ilustrando como usar a API. Pode incluir um exemplo de cdigo. Qualquer texto subsequente que aparea na mesma linha que a @example tag deve ser interpretado como um ttulo para o exemplo. Caso contrrio, a ferramenta de documentao pode indexar os exemplos numericamente.
      ATENO: Jsdoc tem sintaxe diferente. Exemplo...
      @experimental x No tsdoc tem mesma semntica @beta, mas usada por ferramentas que no suportam um @alpha no estgio de lanamento. Exemplo...
      @inherit Doc @inheritDoc Esta tag inline usada para gerar automaticamente a documentao de um item de API, copiando-o de outro item de API. O parmetro de tag inline contm uma referncia ao outro item, que pode ser uma classe no relacionada ou at mesmo uma importao de um pacote NPM separado.
      ATENO: A sintaxe de jsdoc diferente da tsdoc. Exemplo...
      @internal x Designa que um item de API no deve ser usado por desenvolvedores terceirizados. O ferramental pode cortar a declarao de uma liberao pblica. Em algumas implementaes, certos pacotes designados podem consumir itens internos da API, por exemplo, porque os pacotes so componentes do mesmo produto. Exemplo...
      @label x A tag {@label} inline usada para rotular uma declarao, de forma que ela possa ser referenciada usando um seletor na notao de referncia da declarao TSDoc. Exemplo...
      @link @link A tag {@link} inline usada para criar hiperlinks para outras pginas em um sistema de documentao ou URLs gerais da Internet. Em particular, ele oferece suporte a expresses para fazer referncia a itens de API. Exemplo...
      @override @override Este modificador tem semntica semelhante override palavra - chave em C# ou Java. Para uma funo de membro ou propriedade, indica explicitamente que esta definio est substituindo (ou seja, redefinindo) a definio herdada da classe base. A definio da classe base normalmente seria marcada como virtual. Um instrumento de documentao pode impor que as @virtual, @override, e / ou @sealed modificadores so aplicados de forma consistente, mas isto no exigido pela norma TSDoc. Exemplo...
      @package Documentation x Usado para indicar um comentrio de documento que descreve um pacote NPM inteiro (em oposio a um item de API individual pertencente a esse pacote). O comentrio @packageDocumentation encontrado no arquivo*.d.ts que atua como o ponto de entrada para o pacote e deve ser o primeiro /**comentrio encontrado nesse arquivo. Um comentrio contendo uma tag @packageDocumentation nunca deve ser usado para descrever um item individual da API. Exemplo...
      @param @param Usado para documentar um parmetro de funo. A tag @param seguida por um nome de parmetro, seguido por um hfen, seguido por uma descrio.
      ATENO: O jsdoc tem descrio mais detalhada porque javascript no tem tipo. Exemplo...
      @private Remarks @private Inicia uma seo de contedo de documentao adicional que no se destina ao pblico. Uma ferramenta deve omitir esta seo inteira do site de referncia daAPI, o arquivo *.d.ts gerado e quaisquer outras sadas que incorporem o contedo. Exemplo...
      @public @public Designa que o estgio de lanamento de um item de API pblico. Ele foi oficialmente lanado para desenvolvedores terceirizados e sua assinatura garantida como estvel (por exemplo, seguindo as regras de controle de verso semntica). Exemplo...
      @readonly @readonly Esta tag modificadora indica que um item de API deve ser documentado como sendo somente leitura, mesmo se o sistema de tipo TypeScript puder indicar o contrrio. Por exemplo, suponha que uma propriedade de classe tenha uma funo setter que sempre lana uma exceo explicando que a propriedade no pode ser atribuda; nesta situao, a tag @readonly pode ser adicionado para que a propriedade seja mostrada como somente leitura na documentao. Exemplo...
      @remarks x A documentao principal de um item de API separada em uma breve seo de resumo, opcionalmente seguida por uma seo de comentrios mais detalhada. Em um site de documentao, as pginas de ndice (por exemplo, mostrando os membros de uma classe) mostraro apenas os breves resumos, enquanto as pginas de detalhes (por exemplo, descrevendo um nico membro) mostraro o resumo seguido pelos comentrios. A tag @remarks de bloco termina a seo de resumo e comea a seo de comentrios para um comentrio de documento. Exemplo...
      @returns @returns Usado para documentar o valor de retorno de uma funo. Exemplo...
      @sealed x Este modificador tem semntica semelhante sealed palavra - chave em C # ou Java. Para uma classe, indica que as subclasses no devem herdar da classe. Para uma funo ou propriedade de membro, indica que as subclasses no devem substituir (ou seja, redefinir) o membro. Um instrumento de documentao pode impor que as @virtual, @override, e / ou @sealed modificadores so aplicados de forma consistente, mas isto no exigido pela norma TSDoc. Exemplo...
      @see @see Usado para construir uma lista de referncias a um item de API ou outro recurso que pode estar relacionado ao item atual.
      Observao: o JSDoc tenta criar um hiperlink automtico para o texto imediatamente aps @see. Como isso ambguo com texto sem formatao, o TSDoc exige uma {@link}marca explcita para fazer hiperlinks. Exemplo...
      @throws @throws Usado para documentar um tipo de exceo que pode ser lanado por uma funo ou propriedade.
      Um bloco @throws separado deve ser usado para documentar cada tipo de exceo. Esta tag apenas para fins informativos e no impede que outros tipos sejam lanados. sugerido, mas no obrigatrio, que o bloco @throws comece com uma linha contendo apenas o nome da exceo. Exemplo...
      @type Param x Usado para documentar um parmetro genrico. A tag @typeParam seguida por um nome de parmetro, seguido por um hfen, seguido por uma descrio. O analisador TSDoc reconhece essa sintaxe e a extrai em um n DocParamBlock. Exemplo...
      @virtual @abstract Este modificador tem semntica semelhante palavra-chave virtual em C# ou Java. Para uma funo ou propriedade de membro, indica explicitamente que as subclasses podem substituir (ou seja, redefinir) o membro.
      Um instrumento de documentao pode impor que as @virtual, @override, e / ou @sealed so modificadores aplicados de forma consistente, mas isto no exigido pela norma TSDoc. Exemplo...
      @ignore @ignore No aplicativo TypeDoc essa tag impede que o cdigo subsequente seja documentado. Exemplo...
      x x A tag @category do gerador de documento TypeDoc reconhece essa tag. No encontrei um exemplo prtico para esta tag. Exemplo...
      x @module O programa TypeDoc usa para especificar um comentrio na parte superior de um arquivo de origem para documentar esse arquivo de origem e, opcionalmente, substituir o nome de um ponto de entrada. Consulte a seo Arquivos para obter mais detalhes. Exemplo...

  *

  *

  * designed for use in `{@link}` and `{@inheritDoc}` tags, 
  * but you can also use it in your own custom tags.

  *

  *
  * {@link controls.Button.render | the render() method}
  *
  * If a static and instance member have the same name, we can use a selector to distinguish them:
  *
  * {@link controls.Button.(render:instance) | the render() method}
  *
  * {@link controls.Button.(render:static) | the render() static member}
  *
  * This is also how we refer to the class's constructor:

  * | the toNumberPrimitive() static member}

  * See the TSDoc spec for more details about the "declaration reference" notation.

typedoc package1/index.ts package2/index.ts

typedoc ./ts
